<objective>
Ship the Phase 11 ablation seam — two stateless classes (`NullReranker`, `NullBM25Store`) that satisfy the existing `Reranker` and `BM25Store` protocols structurally so Phase 11's no-rerank and dense-only ablations construct an ablated `Retriever` by adapter swap rather than by behavior toggle in `Retriever.search` (D-08). Phase 5's `Retriever.search` keeps zero conditional branches for ablation — control flow is invariant; the adapters degenerate the corresponding stage.

Purpose: Phase 11 (ABL-01) ships ≥3 ablations including no-rerank and dense-only. If those ablations are implemented via conditional flags in `Retriever.search` (`no_rerank=True`, `dense_only=True`), the hot path branches and the Phase 5 acceptance gate (the canary) loses bite — the ablation gates a different code path than the production path. The adapter-swap pattern preserves the same call sequence with degenerate scoring. This plan cashes that design upfront.

Output: 1 new file (`null_adapters.py`); `__init__.py` extended with two new re-exports; `tests/test_null_adapters.py` xfail markers deleted, four tests pass.

BM25Store Protocol contract (existing, from `packages/docintel-core/src/docintel_core/adapters/protocols.py`, full method list):
- `.name -> str`
- `.add(chunks: list[Chunk], text: list[str]) -> None`

- `.query(query_text: str, k: int) -> list[tuple[str, int, float]]`
- `.verify() -> bool` (post-build integrity check)
- `.last_vocab_size() -> int` (Phase 4 D-13 manifest field source)

Both Protocols are `@runtime_checkable`, so `isinstance(obj, Reranker)` and `isinstance(obj, BM25Store)` structurally check the method set.

NullReranker semantics (per D-08 + RESEARCH Pattern 3 lines 571-588):
- `rerank(query, docs)` returns `[RerankedDoc(doc_id=str(i), text=doc, score=-float(i), original_rank=i) for i, doc in enumerate(docs)]`.
- Negated score means descending sort preserves input order (top of input list — RRF rank 0 — gets score 0, bottom — RRF rank M-1 — gets score -(M-1)).
- The downstream `Retriever._rerank` already sorts the reranker output descending by score (BGEReranker does this internally; StubReranker does this internally). With NullReranker, descending sort over the returned RerankedDocs yields the same input order.

NullBM25Store semantics (per D-08 + RESEARCH Pattern 3 lines 591-615):
- `query(text, k)` returns `[]` always — Phase 5's RRF then runs over dense-only.
- `add/commit/verify/last_vocab_size` are no-ops with stable sentinel returns: `.add` returns `None`, `.commit` returns `"0" * 64` (the sentinel 64-char-zero sha256 string for MANIFEST consistency), `.verify` returns `True`, `.last_vocab_size` returns `0`.
- `.name` returns the string `"null-bm25"`.

Verified test analogs:
- `tests/test_adapters.py::test_stub_reranker_sorted` lines 68-89 — Reranker assertion shape: `[r.doc_id, r.text, r.score, r.original_rank]` on each result.
- `tests/test_index_stores.py::test_bm25s_store_satisfies_protocol` lines 61-68 — `isinstance(store, BM25Store)` runtime check pattern.
